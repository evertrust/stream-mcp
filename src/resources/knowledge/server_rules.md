The operating rules for this Stream MCP. Read these first — they are the conventions every Stream tool obeys. When a rule here conflicts with an assumption you carry from another product, this file wins.

## 1. Names are immutable primary keys — ask, never invent

Every Stream object is keyed by its `name` (CAs, templates, keystores, keys, credentials, identity providers, roles, principals, triggers, EKUs, OCSP/TSA signers, …). The `name` is the primary key and **cannot be changed after creation**. There is no rename endpoint.

- Before creating any object, **ask the user for the exact name.** Never guess or auto-generate one.
- Object identifiers passed to other tools (`ca`, `template_name`, `keystore`, `key`, `serial`) are key references — they must match an existing object exactly. Unknown name → `404` (e.g. `CA-003`, `KEYSTORE-003`, `KEY-003`).
- A bad name on create fails validation, e.g. `KEYSTORE-002` (name regex `[0-9a-zA-Z-_.]+`); a duplicate fails, e.g. `KEYSTORE-004`, `KEY-004`.

## 2. Updates are full-replace: GET → strip → merge → PUT

Stream update endpoints PUT the **entire object** onto the collection root, keyed by the body's `name` (no path param). There is no PATCH. **Omitted optional fields are reset** — a partial body silently wipes whatever you left out.

Always:

1. `get_*` (or `list_*`) the current object.
2. Strip server-computed / read-only fields (`status`, `id`).
3. Merge your one change into the full object.
4. Send the whole thing back via the matching `update_*` tool.

Tools with this contract include `update_keystore`, `update_ca`, `update_credential`, `update_eku`, `update_identity_provider`, `update_principal_info`, `update_trigger`, `upsert_system_configuration`.

Server overrides to know:

- **Secrets are retained when omitted.** PKCS#11 `pin` and credential secrets are kept from the previous record if you don't resend them — only send them to _change_ them.
- **CAs are heavily guarded on PUT** (`update_ca`): on an already-issued CA the server restores `certificate`/`privateKey`/`altPrivateKey` from the prior record and forces `dn = None`. To change a CA's cert/key, use `issue_ca` / `enhance_ca` / `migrate_ca`, **not** `update_ca`.
- Local-identity passwords can NOT be changed via `update_*` — use `reset_local_identity_password`.

## 3. Lists return `[]` — but Stream's 204 means empty _or_ forbidden

List/search **tools** always hand you a plain array (or, for query tools, `{ "results": [...] }`). Under the hood Stream returns **HTTP 204 No Content** for an empty collection, which the tools surface as an empty list.

Critical ambiguity: **204 is also what insufficient permission returns** on a list/search (`asyncLogAndReplyNoContent`). An empty list is therefore indistinguishable from "you lack the `audit` permission." If a list comes back empty when you expected data, suspect a missing permission and call `whoami` to check.

Contrast with single-object reads/writes: those return explicit **403** on permission failure and **404** when the object is missing — never a silent empty.

Note the exception: the query tools (`search_events`, `search_certificates`, `search_ssh_certificates`, `aggregate_*`) **never** 204 — an empty result is `200` with `"results": []`.

## 4. Secrets are write-only and come back redacted

Anything sensitive is sent **in** and never returned **out**. Do not expect to read a secret back, and never log one.

| Secret                                  | Where                                     | On read                                                                |
| --------------------------------------- | ----------------------------------------- | ---------------------------------------------------------------------- |
| PKCS#11 `pin`                           | `create_keystore` / `update_keystore`     | `{}`                                                                   |
| Credential secret (`password`/`secret`) | `create_credential` / `update_credential` | `{}` (send `{"clear":"<secret>"}` to set)                              |
| Local-identity `password`               | generated by server on create / reset     | returned **once in clear**, then never again; `hash` is always omitted |
| Private key material                    | keys, PKCS#12 extraction                  | never returned (redacted)                                              |

Rules: on create, the server may generate the secret itself (e.g. local-identity `password` — a `password` you send is discarded). On update, an omitted secret is **retained**. `hash` is write-only/server-managed and ignored on input. Always start an update from a fresh `get_*` so you don't clobber retained secrets.

## 5. Certificates are PEM-in / rich-object-out

You hand Stream a **PEM string**; Stream hands you back a **rich decoded object** (`dn`, `serial` (hex), `notBefore`/`notAfter`, `keyType`, `signingAlgorithm`, `keyUsages`, thumbprints, `pem`, …). This asymmetry is everywhere a certificate appears:

- `enroll_certificate` takes a PKCS#10 CSR as a plain PEM string → returns the full `X509Certificate` object.
- `revoke_certificate` accepts the cert as PEM (or `serial`+`ca`).
- On a CA object, `certificate` is a PEM string on write and a decoded object on read.

When a tool wants a certificate, give it the literal PEM (`-----BEGIN CERTIFICATE-----\n…`), not a parsed object. Enroll is **key-in, cert-out**: the public key (PKCS#10 for X509, OpenSSH pubkey for SSH) is the only key source — there is **no server-side key generation and no PKCS#12/JKS returned** from enroll.

## 6. Exactly two query DSLs — SEQL and SCQL. Nothing else.

Stream has **two** query languages, both dialects of StreamQL (same operators/date grammar, different field sets). Parse errors come back as `400 STREAMQL-001`.

- **SEQL** (Stream Events Query Language) — the `query` for `search_events`.
  - **Omit `query` (or send `null`) to match all** — an absent query compiles to an empty filter. Do not pass `""`.
  - `exists` (and `within`) apply to `detail.<key>` fields **only**. Call `get_event_dictionary` for the valid `code`/`module`/`status`/`detail.*` literals.
- **SCQL** (Stream Certificates Query Language) — the `query` for `search_certificates`, `search_ssh_certificates`, `aggregate_certificates`, `aggregate_ssh_certificates`.
  - **An empty `query` string `""` is REJECTED** (`400 STREAMQL-001 "Invalid condition: ''"`). To match everything, send a real catch-all predicate: **`id exists`**.

```
search_events:        (omit query → match all)   |  code equals SERVICE-START and status equals failure
search_certificates:  status is valid            |  valid.until before 30 days and status is not revoked
                      id exists   (← match-all, NOT "")
```

There is **NO** other query language in Stream. There is no SDQL/SRQL/SPQL and no HEQL/HCQL/HDQL/HRQL. There is **no discovery, no request-workflow, and no permission query language** — do not reference any of those concepts. (See `stream://knowledge/query-languages` for the full grammar.)

## 7. Revocation requires a reason and is terminal — certs are not deletable

Issued certificates are persistent records. **There is no delete-certificate tool.** Revocation is the only terminal action.

- `revoke_certificate` (X509) **requires a `reason`** — one of exactly 7 `CFRevocationReason` values: `unspecified`, `keyCompromise`, `cACompromise`, `affiliationChanged`, `superseded`, `cessationOfOperation`, `certificateHold`. (`removeFromCRL`, `privilegeWithdrawn`, `aACompromise` are not supported.)
- `revoke_ssh_certificate` (SSH) takes **no reason field**.
- Both are **idempotent**: an already-revoked or expired cert returns `200` with current state, not an error. Both are polymorphic: pass `certificate` PEM **or** `serial`+`ca` (PEM wins; serial and ca are required together). X509 serials are lowercase **hex**; SSH serials are **decimal** strings.
- A managed self-signed root CA cert cannot be revoked (`403 CA-017`).

Check `permissions.revoke` from `get_certificate` / `search_certificates` to know in advance whether a revoke will succeed.

## 8. CSR / generate endpoints return raw PEM; decoders & uploads are multipart

- **CSR generation returns raw PEM**, not JSON: `generate_ca_csr`, `generate_ocsp_signer_csr`, `generate_tsa_signer_csr` return a `-----BEGIN CERTIFICATE REQUEST-----…` PKCS#10 string (`Content-Type: application/pkcs10`). Feed that PEM into the corresponding issue/enroll step.
- **Decoders upload content as multipart and request JSON.** `detect_file`, `decode_x509`, `decode_crl`, `decode_csr`, `extract_pkcs12`, `decode_openssh_pubkey` POST the object content (PEM or base64-DER) as a multipart file under an exact field name; with `Accept: application/json` Stream returns the structured decoded object. `extract_pkcs12` also takes the `password` and its returned `privateKey` is a **secret** (redacted, never logged).
- **CRL upload is multipart** too (`upload_crl` — the CRL file must verify under the CA cert).

## 9. Auth is local (X-API headers) or X509/mTLS only — no OIDC for the MCP

This MCP authenticates to Stream in exactly **two** ways, auto-selected from env (priority mTLS > local; neither set = startup error):

- **Local account** — `X-API-ID` / `X-API-KEY` / `X-API-IDPROV` headers (from `STREAM_API_ID` / `STREAM_API_KEY` / `STREAM_API_IDPROV`, the last defaulting to `local`). Sent on every request; no login/session/token flow.
- **X509 / mTLS** — a client cert (PEM or PKCS#12) is the credential; no `X-API-*` headers.

There is **no OIDC for the MCP's own auth.** OIDC (`OpenId`) exists only as a _managed_ identity-provider config object you can CRUD via `create_identity_provider` / `update_identity_provider` — it is never used to authenticate this MCP. There is no CSRF token and no refresh. Use `whoami` to resolve your identity, effective permissions, and roles before scoped writes. (See `stream://knowledge/authentication`.)
